3. Systemd Service Optimization (Critical Learning):
   - ❌ REWRITING SERVICE FILES = DANGEROUS — can crash containers (confirmed by our crash)
   - ✅ DROP-IN CONFIGS in .service.d/ = SAFE — industry standard pattern
   - Pattern: `mkdir -p /etc/systemd/system/code-interpreter.service.d/` then write `10-hardening.conf`
   - This overrides without touching the original service file
   - Debian 13 Trixie (systemd 257) supports: ProtectSystem=strict, NoNewPrivileges=yes, PrivateTmp=yes
   - Use systemd-analyze security to baseline service security scores

4. Cgroups v2 Resource Management:
   - MemoryHigh = soft limit (throttling, service slows but isn't killed)
   - MemoryMax = hard limit (OOM killer activates above this)
   - Both together = safe resource management (back-pressure before OOM)
   - CPUWeight = proportional sharing (1-10000, default 100)
   - IOWeight = proportional I/O priority (replaces IOSchedulingClass for fine-grained control)

5. I/O Scheduling Correction:
   - ❌ IOSchedulingClass=idle — TOO AGGRESSIVE, starves the service of disk I/O
   - ✅ IOWeight=25-100 — Proportional, only throttles under contention
   - IOWeight=50 for background services, IOWeight=500 for critical services (like envd)

6. Docker/Container Tuning (relevant to our environment):
   - userland-proxy=false → iptables directly (kernel handles, more efficient)
   - TCP: tcp_fastopen=3, somaxconn=65535, tcp_tw_reuse=1
   - CPU pinning: cpuset-cpus for cache locality
   - sysctl for high-throughput: net.core.somaxconn=65535, net.core.netdev_max_backlog=65535

What I learned from "Chat (4).txt" — NexusCompress + Android APK/ZArchiver plugin chat:
- Built NexusCompress algorithm in versions V1/V2/V3/V6.
- V1 problem: weak CA-LZ77 and huge Huffman table overhead.
- V2 problem: better compression but broken round-trip due lazy matching not flushing pending match, losing ~5KB.
- V3 problem: used zlib internally then Huffman re-encoded zlib output; this is double-compression and architecturally cannot beat ZIP.
- V3 benchmark on device_configlist.txt (164,136 bytes):
  7z PPMd Ultra 37,235; LZMA -9 38,440; ZIP -9 43,881; NexusCompress v3 44,621. Nexus v3 ranked last.
- Important compression architecture lesson:
  Bad: Input -> zlib(LZ77+Huffman) -> another Huffman -> output.
  Good target: Input -> dynamic dictionary preprocessing -> custom LZ77 token stream -> range/arithmetic/ANS coder -> output.
- APK-aware compression breakthrough:
  APK files are ZIP archives; opaque recompression wastes potential.
  Better: APK/ZIP -> decompress internal DEFLATE entries -> rebuild as STORED ZIP -> compress whole stored archive with LZMA2 -> restore original APK/ZIP structure on decompression.
  This structure-aware APK preprocessing beat standard raw approaches in the chat.
- Known algorithm bugs to remember:
  1. arle_encode marker conflict: bytes 252-255 conflict with RLE markers; workaround: use simple_rle_encode.
  2. dictionary_v2 lazy matching loses final pending match; fix required: flush pending match at stream end.
  3. BPE preprocessor caused data loss; disable until fully reversible.
  4. Python BWT is slow/O(n log n) and caused timeouts; use only selectively or implement faster suffix array/C extension.
  5. Range coder implementation had carry/infinite-loop issues; needs rigorous tests.
