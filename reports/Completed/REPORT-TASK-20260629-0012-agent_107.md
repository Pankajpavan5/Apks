## 2. Core Technical Concepts Acquired

### Concept A: The eBPF Paradigm Shift in Android Dynamic Analysis
Traditional dynamic analysis of Android APKs relies heavily on `ptrace` (via `strace` or Frida) to hook system calls and intercept execution flows. However, `ptrace` introduces massive context-switching overhead (frequently causing target apps to stutter or crash) and is easily detected by commercial anti-tamper mechanisms.
* **The eBPF Solution:** Modern Android (starting with Android 12, hardened in Android 14) requires eBPF support in the Linux kernel (`CONFIG_BPF_SYSCALL`). eBPF allows reverse engineers to load sandboxed bytecode directly into the kernel space.
* **Near-Zero Overhead:** Because eBPF programs execute directly within the kernel event loop using a specialized JIT compiler, they introduce negligible latency. Target APKs cannot detect eBPF instrumentation via standard user-space root detection techniques.

```text
┌───────────────────────────────────────────────────────────────────────────────┐
│                   TRADITIONAL PTRACE VS. eBPF ARCHITECTURE                    │
└───────────────────────────────────────────────────────────────────────────────┘
  [LEGACY PTRACE / FRIDA]
  User Space:   Target APK ──(Context Switch)──> strace / Frida Hook (High Overhead / Detectable)
  Kernel Space: Syscall Execution Layer

  [MODERN eBPF INSTRUMENTATION]
  User Space:   Target APK (Executes Uninterrupted / Completely Transparent)
  Kernel Space: Syscall Layer ──> BPF Bytecode JIT Hook ──> BPF Ring Buffer (Zero Overhead)
```

### Concept B: Kprobes and Tracepoints for Sysfs & Binder Auditing
The presentation detailed exactly how to write eBPF programs using C/BPF toolchains to monitor privileged hardware daemons (such as the Samsung GOS service analyzed in previous tasks).
* **Monitoring Hardware Interventions (`sys_openat` / `sys_write`):**
  By attaching a kernel probe (`kprobe/sys_openat` and `kprobe/sys_write`), an eBPF program can instantly inspect every file path opened by an application. If a privileged system daemon touches `/sys/class/thermal/` or `/sys/class/kgsl/`, the eBPF map logs the exact process ID (PID), UID, and written byte buffer.
* **Binder IPC Interception (`tracepoint/binder/binder_transaction`):**
  By hooking the kernel's Binder tracepoint, eBPF programs can parse raw Binder transaction codes (`code 1034`, `code 1035`) and destination handles in real-time. This provides incontrovertible proof of when an APK issues display refresh rate clamps or background process purging instructions to `SurfaceFlinger`.

### Concept C: Automating JNI Interception via User-Space Uprobes
While `kprobes` hook kernel functions, `uprobes` (User-space Probes) allow eBPF bytecode to hook functions inside native C/C++ shared libraries (`.so` files) running in user space.
* **Hooking Native Libraries (`libipm.so` / `libtcm.so`):**
  The YouTube presentation demonstrated how to calculate function offsets within an ELF binary using `readelf -Ws`. By attaching a `uprobe` to the calculated offset of `Java_com_samsung_android_game_gos_ipm_IpmWrapJNI_IpmJava_1setPerGameTraining`, the eBPF program intercepts the exact input parameter registers (such as frame draw times and thermal weights) right before the native C++ execution begins.
* **Extracting AI Neural Net Weights:**
  When `libipm.so` invokes `mmap()` to map `ipm_target_model.tflite` into memory, a `uretprobe` (User-space Return Probe) can capture the returned memory address pointer, allowing security tools to dump the raw, uncompressed TensorFlow Lite neural network model directly from kernel space buffers.
